## 2. Concrete Corrections & Quality Refactoring

During development, the AI generated legacy patterns or silent defaults that were corrected to meet the requirements of the project.

### Case 1: Prisma 7 Database Datasource URL
- **What the AI did:** Generated a traditional `schema.prisma` file containing `url = env("DATABASE_URL")` in the `datasource db` block.
- **Why it was wrong:** Prisma 7 introduced breaking changes that deprecate `url` in schema files, moving all migration and introspection database connections to `prisma.config.ts`.
- **How we caught it:** When running `npx prisma migrate dev --name init`, the CLI threw:
  `Error code: P1012 - The datasource property url is no longer supported in schema files.`
- **What we changed:** Edited the `schema.prisma` file to remove the `url` property from the datasource block, relying on the automatically loaded environment variables in `prisma.config.ts`.

### Case 2: Prisma 7 Client Connection (Rust Engine Deprecation)
- **What the AI did:** Instantiated Prisma Client using the standard `new PrismaClient()` without arguments in Node.js.
- **Why it was wrong:** Prisma 7 deprecates the Rust-based sidecar engine (library/binary query engines) by default, switching to a WASM query compiler that requires an explicit database adapter constructor option for standard SQL drivers.
- **How we caught it:** Running the seed script crashed with:
  `PrismaClientConstructorValidationError: Using engine type "client" requires either "adapter" or "accelerateUrl"`
- **What we changed:** Installed `@prisma/adapter-pg` and `pg` packages, and refactored the initialization script to pass the custom pg Pool driver adapter to the `PrismaClient` constructor.

### Case 3: Silent Whitespace Trimming in CSV Ingestion
- **What the AI did:** Implemented an ingestion function `parseCSV` that automatically called `.trim()` on all parsed CSV cells.
- **Why it was wrong:** Trimming cells silently corrects issues like trailing whitespaces in names (e.g. `"Aisha "` to `"Aisha"`) without logging it. This violated Meera's requirement (*"I want to approve anything the app deletes or changes"*), and the PDF requirement (*"A silent guess is a failing answer"*).
- **How we caught it:** During the parser verification CLI run, we observed that Row 13 (`"Aisha "`) did not flag any name anomalies, meaning the whitespace was being corrected silently.
- **What we changed:** Refactored `parseCSV` to return raw, untrimmed strings. The validation engine `validateCSVRows` was then updated to explicitly identify, log, and surface `INCONSISTENT_NAME_WHITESPACE` as an anomaly that the user must approve to resolve.
